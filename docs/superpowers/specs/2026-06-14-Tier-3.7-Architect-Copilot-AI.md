## 1. Promesse cardinale

**Un agent LLM consomme toute la surface MCP analytics du repo et produit une interprétation narrative structurée (Constat → Causes → Recommandations → Confiance) à 3 profondeurs (`quick` / `standard` / `deep`), surfacée comme panel UI + endpoint REST + skill Claude réutilisable — pure couche de synthèse, zéro nouvelle analytique.**

## 2. Goal

Surfacer dans l'UI web un panel "Architect Tour" qui, sur demande, lance un agent LLM côté serveur (`tour-worker.mjs`) consommant les outils MCP existants pour produire un rapport narratif (markdown + JSON structuré) sur l'état architectural d'un repo. Le panel affiche le rapport en streaming (SSE). Endpoint REST `POST /tour` côté `docker-server-tour.mjs`. Skill Claude `.claude/skills/gitnexus-architect.md` réutilisable hors-UI.

Succès = depuis l'UI, l'utilisateur sélectionne un repo + profondeur, clique "Generate tour", voit le rapport s'écrire en stream avec 4 sections (Constat / Causes / Recommandations / Confiance) et un score de confiance global ; et un agent Claude externe peut invoquer le même tour via le skill MCP.

## 3. Implementation steps

1. **MCP analytics inventory gate** : valider que les 12 tools `gitnexus_*` exposés par `mcp-server/server.mjs` couvrent les 9 endpoints requis (entropy, churn, coupling, growth, lifespan, ownership, dissonance, semantic_labels, similarity). Smoke `node mcp-server/smoke.mjs`. Si gap → bloquer Phase A.
2. **Tour-worker skeleton (`tour-worker.mjs`)** : nouveau worker Node zéro-dep dans `gitnexus-server`, HTTP :4749, endpoint `POST /tour` accepte `{ repo, depth, model? }`, retourne SSE stream `event: section`. Spawn LLM via `GITNEXUS_API_KEY`/`GITNEXUS_LLM_BASE_URL` env (même pattern que wiki-worker Tier 2.5).
3. **Prompt template structuré** : 4 sections fixes (Constat, Causes, Recommandations, Confiance), schéma JSON output forcé via tool-use ou structured-output. Profondeurs : `quick` = entropy + ownership ; `standard` = + dissonance + churn + similarity ; `deep` = + coupling + growth + lifespan + semantic-labels.
4. **MCP client embedded** : le worker parle stdio JSON-RPC 2.0 au `mcp-server/server.mjs` (sibling process spawn) pour invoquer les tools. Réutilise la surface publique MCP, zéro nouveau couplage à l'API REST sous-jacente.
5. **Docker-server-tour.mjs (web container)** : nouveau handler dans `gitnexus-web` qui proxy `POST /tour` → `http://gitnexus:4749/tour`, stream SSE 1:1. Monte via `docker-server.mjs > handleTourRoute`.
6. **TourPanel.tsx (UI)** : nouveau composant React + state hook `useTour({ repo, depth })`, EventSource sur `/tour`, rendu sections incrémentales avec marked.js (mêmes deps que WikiPanel).
7. **Skill Claude `.claude/skills/gitnexus-architect.md`** : description pour invocation MCP hors-UI, exemples d'usage agentique (`/architect tour --repo=X --depth=deep`).
8. **Cache TTL 300s** : par `(repo, depth, snapshotId)` — les snapshots passés sont immuables, le tour peut être cache infiniment ; le snapshot HEAD se cache 5min.
9. **Doc + INVENTORY update** : entrée dans `## ✅ Déjà livré` + INVENTORY ligne MCP tool count (+1 tool `gitnexus_tour`).

## 6. Tests E2E

1. **Quick tour smoke** : `POST /tour { repo: "test-repo", depth: "quick" }` → SSE émet 4 sections (Constat/Causes/Reco/Confiance) en <30s, confidence ∈ [0,1].
2. **Deep tour complétude** : `depth: "deep"` invoque les 9 endpoints MCP (vérifier `toolsCalls` dans event `done`), génère >2KB markdown.
3. **Cache hit** : 2 appels successifs `(repo, depth)` identiques → 2ème en <100ms, `event: done { cacheHit: true }`.
4. **Skill Claude MCP** : invocation `gitnexus_tour` via `node mcp-server/smoke.mjs --tool=tour` → JSON rapport retourné, sections présentes.
5. **Mode dégradé LLM** : `GITNEXUS_API_KEY` absent → `POST /tour` retourne 503 + UI affiche "LLM not configured, see docs" (jamais crash).

## 7. Iron Rules / Patterns Mycelium synergy

**Iron Rules ELYSIUM** (frontmatter SIGIL companion à graver post-livraison) :

- **COPILOT-1** : *Tour est synthèse pure, jamais nouvelle analytique* — toute nouvelle métrique doit naître ailleurs (Tier 2.5/3.x) puis être consommée par tour. Anti-pattern : ajouter une heuristique métier dans le worker. **Implémentation Task A1** : `upstream/docker-server-copilot-core.mjs > inventoryMCPTools()` — fonction de synthèse pure (mappe les 9 endpoints requis aux MCP tools existants, zéro nouvelle métrique).
- **COPILOT-2** : *MCP est l'unique surface d'accès aux données* — le worker ne parle JAMAIS directement à `/entropy` REST ; toujours via `gitnexus_*` MCP tools. Évite couplage à l'API privée. **Implémentation Task A1** : MCP tool `gitnexus_copilot_inventory` (cf `mcp-server/server.mjs`) expose le gate verdict via la surface MCP (jamais via fetch direct sur `/copilot/mcp-inventory`).
- **COPILOT-3** : *Confidence score obligatoire* — toute section narrative doit déclarer son niveau de confiance ∈ [0,1] basé sur sample size / récence / completeness. Aligne avec ELYSIUM Iron Rule MEM-3 (preuve avant assertion).
- **COPILOT-4** : *Cache by snapshot, not by clock* — la clé cache inclut le `snapshotId` HEAD au moment de la génération, pas le timestamp. Snapshots passés = cache ∞.
- **COPILOT-5** : *Mode dégradé sans LLM = explicite* — endpoint retourne 503 avec instructions, jamais 500 silencieux.

**Mycelium synergy (cross-domain ELYSIUM)** :

- **Belt Market BLT supply chain** : le tour peut, en mode `deep`, consommer le hash-chained ledger Belt Market pour annoter les recommandations d'un coût économique (ex: "refactoriser oauth.ts coûte ~3 BLT vs 7 BLT dette technique accumulée").
- **Cluster ops audit hash chain** (SIGIL-1255 sortance + SIGIL-1252 coordination) : le tour peut citer la dette de coverage (meta_score sortance) et la merge survival rate du repo comme inputs additionnels au scoring confiance.
- **ASTKG forge feed** : le rapport narratif final est append-only ingéré dans `data/governance/architect_tour_audit.jsonl` (hash-chained, source pour SIGIL-1547 External Knowledge Atlas).
- **factory_event_sourced_append_only.mdc** : pattern Mycelium directement appliqué — chaque tour run = un event sourced, immutable, replayable.
