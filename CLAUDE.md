# gitnexus — Deployment & Time-Travel Analytics

This folder is the **deployment + analytics extension** on top of
upstream [gitnexus/gitnexus](https://github.com/abhigyanpatwari/gitnexus)
(pinned to `v1.6.3`). Our deltas live in two places: the few files we
own at this folder's root (`Dockerfile.cli`, `docker-compose.yml`,
`scripts/`), and a serialized diff of everything we patch in `upstream/`.

The general workspace rules in `../CLAUDE.md` apply. The rules below
are gitnexus-specific.

## Source-of-truth references (read before non-trivial work)

| File | What it tells you |
|---|---|
| `ROADMAP.md` | Tiers, what's planned vs done. Update it when you ship. |
| `INVENTORY.md` | Full state-of-the-deployment: upstream features, our additions, distance from upstream. Update on every release or new feature. |
| `patches/upstream-all.diff` | Every modification we apply to `upstream/`. Regenerated by hand, see below. |
| `patches/README.md` | How to re-apply patches on a fresh upstream clone. |

## When you ship a feature here

In addition to the workspace-level rule (update ROADMAP, update
INVENTORY, add tests if applicable):

1. **Regenerate `patches/upstream-all.diff`** if you edited anything
   inside `upstream/`. The diff is what survives — un-staged upstream
   edits are invisible to anyone who re-clones this folder.

   ```powershell
   cd upstream
   git add -N .
   git diff HEAD > ../patches/upstream-all.diff
   git reset
   cd ..
   ```

2. **Update `ROADMAP.md`** :
   - Mark the Tier item ✅
   - Add a row to the "Déjà livré" table at the top with endpoint/component pointers.

3. **Update `INVENTORY.md`** :
   - "Partie B — Nos ajouts" or wherever the relevant section is.
   - The point is that someone reading INVENTORY alone gets the truth.

4. **Commit at the gitnexus folder root** (this folder is the git repo).
   - Include all three: source files (top level only — `upstream/` is
     `.gitignore`-d), `patches/upstream-all.diff`, and the doc updates.
   - Write a body that names the new endpoints / components.

## Tests & CI/CD

State of the world (2026-05-26):

- **No automated test suite** at the gitnexus deployment level. Smoke
  tests are done manually by curl-ing the new endpoints after rebuild.
- **No CI pipeline.** The closest we have is `docker compose build` —
  if that fails, the patch is broken.
- **Patch script `scripts/patch-lbug-staleness.mjs` IS self-validating**:
  the build fails loudly if upstream's `ensureLbugInitialized` shape
  changed. That's our only automated guard.

So the rule for now: when you add a backend endpoint or a new docker-
server module, **run `docker compose build gitnexus-web` and verify
the new endpoint responds to `curl`** before declaring the work done.
If you add a test suite or CI later, codify this requirement here.

## After-restart smoke checks (canonical commands)

```bash
# Curl every backend module so we catch regressions in routing wiring.
for ep in repos snapshots churn coupling growth lifespan entropy ownership semantic-labels; do
  curl -s -o /dev/null -w "$ep: HTTP %{http_code}\n" \
    "http://localhost:4173/$ep?repo=hmm_studio"
done
```

(Add new endpoints to that loop when you ship them.)
